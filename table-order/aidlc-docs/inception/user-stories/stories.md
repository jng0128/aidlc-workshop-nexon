# 테이블오더 서비스 - 사용자 스토리

## 스토리 구성 방식
- **분류**: Persona-Based (고객 / 관리자)
- **세분화**: 중간 단위 (주요 기능별)
- **수용 기준**: 혼합 (복잡한 스토리는 Given-When-Then, 단순한 스토리는 체크리스트)
- **우선순위**: Must (필수) / Should (권장) / Could (선택)

---

# 🧑‍💼 고객 스토리 (Customer Stories)

## CS-01: 테이블 자동 로그인
**As a** 고객  
**I want** 테이블에 앉으면 별도 로그인 없이 바로 메뉴 화면을 볼 수 있기를  
**So that** 대기 시간 없이 즉시 주문을 시작할 수 있다

**Priority**: Must

**Acceptance Criteria**:
- Given 관리자가 태블릿에 매장 식별자, 테이블 번호, 비밀번호를 1회 설정했을 때
- When 태블릿 브라우저를 열면
- Then 저장된 정보로 자동 로그인되어 메뉴 화면이 표시된다

- Given 자동 로그인에 실패했을 때
- When 로그인 오류가 발생하면
- Then 초기 설정 화면이 표시된다

**의존성**: 없음 (독립적)

---

## CS-02: 메뉴 조회 및 탐색
**As a** 고객  
**I want** 카테고리별로 메뉴를 탐색하고 상세 정보를 확인할 수 있기를  
**So that** 원하는 메뉴를 쉽게 찾아 선택할 수 있다

**Priority**: Must

**Acceptance Criteria**:
- Given 메뉴 화면에 있을 때
- When 카테고리를 선택하면
- Then 해당 카테고리의 메뉴 목록이 카드 형태로 표시된다

- Given 메뉴 카드가 표시될 때
- When 카드를 확인하면
- Then 메뉴명, 가격, 설명, 이미지가 표시된다

추가 체크리스트:
- [ ] 카테고리 간 빠른 이동 가능
- [ ] 터치 친화적 버튼 크기 (최소 44x44px)
- [ ] 카드 형태 레이아웃 적용

**의존성**: CS-01 (로그인 후 접근 가능)

---

## CS-03: 장바구니 관리
**As a** 고객  
**I want** 선택한 메뉴를 장바구니에 담고 수량을 조절할 수 있기를  
**So that** 주문 전에 원하는 메뉴와 수량을 정리할 수 있다

**Priority**: Must

**Acceptance Criteria**:
- Given 메뉴 화면에서 메뉴를 선택했을 때
- When 장바구니에 추가하면
- Then 장바구니에 해당 메뉴가 추가되고 총 금액이 업데이트된다

- Given 장바구니에 메뉴가 있을 때
- When 수량을 변경하면
- Then 수량과 총 금액이 실시간으로 재계산된다

추가 체크리스트:
- [ ] 메뉴 삭제 가능
- [ ] 장바구니 전체 비우기 가능
- [ ] 페이지 새로고침 시에도 장바구니 유지 (로컬 저장)
- [ ] 서버 전송은 주문 확정 시에만 수행

**의존성**: CS-02 (메뉴 조회 후 담기 가능)

---

## CS-04: 주문 생성
**As a** 고객  
**I want** 장바구니의 메뉴를 확인하고 주문을 확정할 수 있기를  
**So that** 실제 주문이 매장에 전달된다

**Priority**: Must

**Acceptance Criteria**:
- Given 장바구니에 1개 이상의 메뉴가 있을 때
- When 주문 확정 버튼을 누르면
- Then 주문이 서버에 전송되고 주문 번호가 표시된다

- Given 주문이 성공했을 때
- When 5초가 경과하면
- Then 메뉴 화면으로 자동 리다이렉트되고 장바구니가 비워진다

- Given 주문이 실패했을 때
- When 에러가 발생하면
- Then 에러 메시지가 표시되고 장바구니는 유지된다

**의존성**: CS-03 (장바구니에 메뉴가 있어야 주문 가능)

---

## CS-05: 주문 내역 조회
**As a** 고객  
**I want** 현재 테이블 세션의 주문 내역을 확인할 수 있기를  
**So that** 지금까지 주문한 메뉴와 금액, 상태를 파악할 수 있다

**Priority**: Must

**Acceptance Criteria**:
- Given 주문 내역 화면에 접근했을 때
- When 현재 세션의 주문이 있으면
- Then 주문 시간 순으로 주문 목록이 표시된다

추가 체크리스트:
- [ ] 주문별 정보 표시: 주문 번호, 시각, 메뉴/수량, 금액, 상태
- [ ] 주문 상태 표시: 대기중/준비중/완료/취소
- [ ] 현재 테이블 세션 주문만 표시 (이전 세션 제외)
- [ ] 이용 완료 처리된 주문 제외

**의존성**: CS-04 (주문이 있어야 내역 조회 가능)

# 🔧 관리자 스토리 (Admin Stories)

## AS-01: 매장 관리자 인증
**As a** 매장 관리자  
**I want** 매장 식별자와 계정 정보로 로그인할 수 있기를  
**So that** 매장 관리 시스템에 안전하게 접근할 수 있다

**Priority**: Must

**Acceptance Criteria**:
- Given 관리자 로그인 화면에서
- When 올바른 매장 식별자, 사용자명, 비밀번호를 입력하면
- Then JWT 토큰이 발급되고 대시보드로 이동한다

- Given 로그인에 성공한 후
- When 16시간이 경과하면
- Then 자동으로 로그아웃되고 로그인 화면으로 이동한다

추가 체크리스트:
- [ ] 브라우저 새로고침 시 세션 유지
- [ ] 비밀번호 bcrypt 해싱 저장
- [ ] 로그인 시도 제한
- [ ] 시드 데이터로 초기 관리자 계정 생성

**의존성**: 없음 (독립적)

---

## AS-02: 실시간 주문 모니터링
**As a** 매장 관리자  
**I want** 들어오는 주문을 실시간으로 테이블별 대시보드에서 확인할 수 있기를  
**So that** 주문을 놓치지 않고 빠르게 처리할 수 있다

**Priority**: Must

**Acceptance Criteria**:
- Given 관리자 대시보드에 있을 때
- When 고객이 새 주문을 생성하면
- Then 2초 이내에 해당 테이블 카드에 주문이 표시된다

- Given 테이블 카드에 주문이 표시될 때
- When 카드를 클릭하면
- Then 전체 메뉴 목록 상세 보기가 표시된다

추가 체크리스트:
- [ ] SSE 기반 실시간 업데이트
- [ ] 그리드 레이아웃: 테이블별 카드 형태
- [ ] 각 카드에 총 주문액 및 최신 주문 미리보기
- [ ] 신규 주문 시각적 강조 (색상/애니메이션)
- [ ] 테이블별 필터링 기능

**의존성**: AS-01 (로그인 후 접근 가능)

---

## AS-03: 주문 상태 관리
**As a** 매장 관리자  
**I want** 주문 상태를 변경할 수 있기를 (대기중 → 준비중 → 완료, 또는 취소)  
**So that** 주문 처리 진행 상황을 관리하고 고객에게 알릴 수 있다

**Priority**: Must

**Acceptance Criteria**:
- Given 대기중 상태의 주문이 있을 때
- When 준비중으로 상태를 변경하면
- Then 주문 상태가 업데이트되고 고객 화면에도 반영된다

- Given 대기중 또는 준비중 상태의 주문이 있을 때
- When 취소를 선택하면
- Then 주문 상태가 취소로 변경된다

- Given 완료 상태의 주문이 있을 때
- When 취소를 시도하면
- Then 취소가 불가능하다는 메시지가 표시된다

**의존성**: AS-02 (모니터링 화면에서 상태 변경)

---

## AS-04: 테이블 관리 - 초기 설정
**As a** 매장 관리자  
**I want** 테이블 번호와 비밀번호를 설정할 수 있기를  
**So that** 고객 태블릿이 자동 로그인할 수 있도록 준비할 수 있다

**Priority**: Must

**Acceptance Criteria**:
- [ ] 테이블 번호 설정 가능
- [ ] 테이블 비밀번호 설정 가능
- [ ] 설정 완료 시 16시간 세션 생성
- [ ] 설정 정보 저장 및 자동 로그인 활성화

**의존성**: AS-01 (관리자 인증 후 접근)

---

## AS-05: 테이블 관리 - 주문 삭제
**As a** 매장 관리자  
**I want** 특정 주문을 삭제할 수 있기를  
**So that** 잘못된 주문을 직권으로 수정할 수 있다

**Priority**: Must

**Acceptance Criteria**:
- Given 테이블에 주문이 있을 때
- When 주문 삭제 버튼을 누르면
- Then 확인 팝업이 표시된다

- Given 확인 팝업에서 확인을 누르면
- When 삭제가 처리되면
- Then 주문이 즉시 삭제되고 테이블 총 주문액이 재계산된다

**의존성**: AS-02 (모니터링 화면에서 접근)

---

## AS-06: 테이블 관리 - 이용 완료 처리
**As a** 매장 관리자  
**I want** 고객 퇴장 시 테이블 이용 완료 처리를 할 수 있기를  
**So that** 새 고객이 이전 주문 없이 깨끗한 상태에서 시작할 수 있다

**Priority**: Must

**Acceptance Criteria**:
- Given 테이블에 활성 세션이 있을 때
- When 이용 완료 버튼을 누르면
- Then 확인 팝업이 표시된다

- Given 확인 팝업에서 확인을 누르면
- When 이용 완료가 처리되면
- Then 현재 주문 내역이 과거 이력으로 이동하고, 테이블 주문 목록과 총액이 0으로 리셋된다

**의존성**: AS-02 (모니터링 화면에서 접근)

---

## AS-07: 테이블 관리 - 과거 주문 내역 조회
**As a** 매장 관리자  
**I want** 테이블별 과거 주문 내역을 조회할 수 있기를  
**So that** 이전 고객의 주문 이력을 확인할 수 있다

**Priority**: Should

**Acceptance Criteria**:
- [ ] "과거 내역" 버튼으로 접근
- [ ] 테이블별 과거 주문 목록 표시 (시간 역순)
- [ ] 각 주문 정보: 주문 번호, 시각, 메뉴 목록, 총 금액, 이용 완료 시각
- [ ] 날짜 필터링 기능
- [ ] "닫기" 버튼으로 대시보드 복귀

**의존성**: AS-06 (이용 완료 처리된 주문이 있어야 조회 가능)

---

## AS-08: 메뉴 관리
**As a** 매장 관리자  
**I want** 메뉴를 등록, 수정, 삭제하고 노출 순서를 조정할 수 있기를  
**So that** 매장 메뉴를 동적으로 관리할 수 있다

**Priority**: Must

**Acceptance Criteria**:
- Given 메뉴 관리 화면에서
- When 새 메뉴를 등록하면
- Then 메뉴명, 가격, 설명, 카테고리, 이미지 URL이 저장되고 고객 화면에 반영된다

추가 체크리스트:
- [ ] 메뉴 CRUD (등록/조회/수정/삭제)
- [ ] 카테고리별 조회
- [ ] 메뉴 노출 순서 조정 (드래그 또는 순서 번호)
- [ ] 필수 필드 검증 (메뉴명, 가격, 카테고리)
- [ ] 가격 범위 검증 (0 이상)

**의존성**: AS-01 (관리자 인증 후 접근), AS-09 (카테고리가 존재해야 메뉴 등록 가능)

---

## AS-09: 카테고리 관리
**As a** 매장 관리자  
**I want** 메뉴 카테고리를 추가, 수정, 삭제할 수 있기를  
**So that** 메뉴를 체계적으로 분류하고 관리할 수 있다

**Priority**: Must

**Acceptance Criteria**:
- Given 카테고리 관리 화면에서
- When 새 카테고리를 추가하면
- Then 카테고리가 생성되고 메뉴 등록 시 선택 가능해진다

- Given 기존 카테고리를 수정할 때
- When 카테고리명을 변경하면
- Then 해당 카테고리에 속한 모든 메뉴의 카테고리도 동일하게 업데이트된다

- Given 카테고리에 메뉴가 존재할 때
- When 카테고리 삭제를 시도하면
- Then 삭제가 거부되고 "카테고리 내 메뉴가 존재하여 삭제할 수 없습니다" 메시지가 표시된다

- Given 카테고리에 메뉴가 없을 때
- When 카테고리 삭제를 시도하면
- Then 확인 팝업 후 카테고리가 삭제된다

추가 체크리스트:
- [ ] 카테고리 CRUD (추가/조회/수정/삭제)
- [ ] 카테고리 수정 시 하위 메뉴 카테고리 자동 동기화 (cascade update)
- [ ] 메뉴가 있는 카테고리 삭제 방지 (delete protection)
- [ ] 카테고리명 중복 검증
- [ ] 카테고리 노출 순서 조정

**의존성**: AS-01 (관리자 인증 후 접근)

# 📊 스토리 요약

## 우선순위별 분류

| Priority | 스토리 |
|----------|--------|
| **Must** | CS-01, CS-02, CS-03, CS-04, CS-05, AS-01, AS-02, AS-03, AS-04, AS-05, AS-06, AS-08, AS-09 |
| **Should** | AS-07 |

## 의존성 다이어그램

```
고객 스토리:
CS-01 (자동 로그인) → CS-02 (메뉴 조회) → CS-03 (장바구니) → CS-04 (주문 생성) → CS-05 (주문 내역)

관리자 스토리:
AS-01 (인증) → AS-02 (주문 모니터링) → AS-03 (상태 관리)
                                      → AS-05 (주문 삭제)
                                      → AS-06 (이용 완료) → AS-07 (과거 내역)
AS-01 (인증) → AS-04 (테이블 설정)
AS-01 (인증) → AS-09 (카테고리 관리) → AS-08 (메뉴 관리)
```

## INVEST 기준 검증

| 기준 | 충족 여부 | 설명 |
|------|:---------:|------|
| **Independent** | ✅ | 각 스토리는 독립적으로 구현 가능 (의존성은 순서만 의미) |
| **Negotiable** | ✅ | 수용 기준 내에서 구현 방식 협상 가능 |
| **Valuable** | ✅ | 각 스토리가 사용자에게 직접적 가치 제공 |
| **Estimable** | ✅ | 중간 단위로 작업량 추정 가능 |
| **Small** | ✅ | 각 스토리가 1~3일 내 구현 가능한 크기 |
| **Testable** | ✅ | 명확한 수용 기준으로 테스트 가능 |
